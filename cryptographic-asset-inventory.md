# Cryptographic Asset Inventory — QuantumShield Scanner

| Cryptographic Asset Type | Element | Description |
|---|---|---|
| Algorithms | Name | AES-128-GCM refers to the AES algorithm with a 128-bit key in Galois/Counter Mode (GCM). AES-256-GCM refers to AES with a 256-bit key in GCM. ChaCha20-Poly1305 is a 256-bit AEAD cipher. RSA-2048/4096 is an asymmetric algorithm. ECDHE uses Elliptic Curve Diffie-Hellman Ephemeral. ML-KEM (Kyber) is a NIST PQC key encapsulation mechanism. ML-DSA (Dilithium) is a NIST PQC digital signature algorithm. SLH-DSA (SPHINCS+) is a NIST PQC hash-based signature. FN-DSA (FALCON) is a NIST PQC lattice-based signature. |
| Algorithms | Asset Type | "algorithm" — all cryptographic primitives detected during scanning are classified as algorithm assets in the CBOM. |
| Algorithms | Primitive | Key Exchange: RSA, DHE, ECDHE, X25519, X448, ML-KEM (PQC). Digital Signature: RSA, ECDSA, EdDSA, ML-DSA, SLH-DSA, FN-DSA. Symmetric Cipher: AES-256-GCM, AES-128-GCM, ChaCha20-Poly1305, AES-CBC, 3DES. Hash: SHA-256, SHA-384, SHA-512. Hybrid: X25519+ML-KEM-768. |
| Algorithms | Mode | GCM (Galois/Counter Mode) used with AES-128 and AES-256 for authenticated encryption. CBC (Cipher Block Chaining) used with AES and 3DES. CCM (Counter with CBC-MAC) used with AES-128. Poly1305 used as authenticator with ChaCha20. |
| Algorithms | Crypto Functions | Key generation, encryption, decryption, and authentication tag generation for AES-GCM, AES-CBC, and ChaCha20-Poly1305. Key encapsulation and decapsulation for ML-KEM-512/768/1024. Digital signing and verification for RSA, ECDSA, EdDSA, ML-DSA, SLH-DSA, FN-DSA. Hashing via SHA-256 for CBOM integrity verification. |
| Algorithms | Classical security level | AES-256: 256-bit (128-bit effective post-quantum, still strong). AES-128: 128-bit (64-bit effective post-quantum, weakened). ChaCha20: 256-bit (128-bit effective, safe). RSA-2048: ~112-bit classical (broken by Shor's). ECDHE P-256: ~128-bit classical (broken by Shor's). ML-KEM-768: NIST Level 3 (192-bit post-quantum). SHA-256: 128-bit collision resistance. |
| Algorithms | OID | AES-128-GCM: 2.16.840.1.101.3.4.1.6. AES-256-GCM: 2.16.840.1.101.3.4.1.46. RSA: 1.2.840.113549.1.1.1. ECDSA-SHA256: 1.2.840.10045.4.3.2. SHA-256: 2.16.840.1.101.3.4.2.1. SHA-384: 2.16.840.1.101.3.4.2.2. ML-KEM: per NIST FIPS 203. ML-DSA: per NIST FIPS 204. SLH-DSA: per NIST FIPS 205. FN-DSA: per NIST FIPS 206. |
| Algorithms | List | RSA, DHE, ECDHE, ECDH, X25519, X448, PSK, ML-KEM-512, ML-KEM-768, ML-KEM-1024, X25519+Kyber768, ECDSA, EdDSA, DSS, ML-DSA, SLH-DSA, FN-DSA, AES-128-GCM, AES-128-CBC, AES-256-GCM, AES-256-CBC, AES-192-GCM, ChaCha20-Poly1305, 3DES-EDE-CBC, RC4-128, DES, SHA-256, SHA-384, SHA-512, SHA-1, MD5. Each assessed for post-quantum readiness per NIST PQC standards. |
| Keys | Name | Key names are derived from the Common Name (CN) in X.509 certificates extracted during TLS handshake scans (e.g., www.example.com). Each key is uniquely identified within the CBOM record. |
| Keys | Asset Type | "key" — all key material extracted from scanned endpoints is classified as a key asset in the CBOM. |
| Keys | id | Unique key identifier derived from the X.509 certificate serial number and SHA fingerprint extracted during TLS scans. |
| Keys | state | Determined from certificate validity: Active (current date within validity window), Expired (past Not Valid After date), or Revoked. Inferred from X.509 certificate fields during scanning. |
| Keys | size | RSA: 2048-bit or 4096-bit. ECDSA P-256: 256-bit. ECDSA P-384: 384-bit. Ed25519: 256-bit. ML-KEM-512: NIST Level 1. ML-KEM-768: NIST Level 3. ML-KEM-1024: NIST Level 5. AES symmetric keys: 128-bit, 192-bit, or 256-bit. |
| Keys | Creation Date | The date the key was created, extracted from the "Not Valid Before" field of the X.509 certificate parsed during TLS scans. |
| Keys | Activation Date | The date the key became operational, corresponding to the "Not Valid Before" date of the X.509 certificate. |
| Protocols | Name | TLS (Transport Layer Security) — primary protocol scanned. IPSec IKEv2 — VPN protocol (UDP 500/4500). OpenVPN (TCP/UDP 1194). WireGuard (UDP 51820). HTTPS — HTTP over TLS. |
| Protocols | Asset Type | "protocol" — all protocol configurations detected on target endpoints are classified as protocol assets in the CBOM. |
| Protocols | Version | TLS 1.0 (deprecated, flagged Critical). TLS 1.1 (deprecated, flagged Critical). TLS 1.2 (supported, commonly detected). TLS 1.3 (recommended, supports PQC key exchange negotiation). All four versions probed via multi-version ClientHello handshakes. |
| Protocols | Cipher Suites | Each cipher suite is decomposed into Key Exchange (ECDHE, RSA, DHE, X25519, ML-KEM-768), Bulk Encryption (AES-256-GCM, AES-128-CBC, ChaCha20-Poly1305), MAC/Hash (SHA-256, SHA-384), and Authentication (RSA, ECDSA). Examples: TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256, TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384. |
| Protocols | OID | OIDs associated with cipher suites are mapped using the IANA TLS Cipher Suite Registry. Each cipher suite ID is mapped to its standardized components for CBOM decomposition. |
| Certificates | Name | The certificate Common Name (CN), referring to the subject domain or entity (e.g., www.pnbindia.in). Extracted from the X.509 Subject field during TLS handshake. |
| Certificates | Asset Type | "certificate" — all X.509 certificates extracted from scanned endpoints are classified as certificate assets in the CBOM. |
| Certificates | Subject Name | The Distinguished Name (DN) of the entity the certificate represents. Contains Organization (O), Common Name (CN), Country (C), and domain name. Extracted from the X.509 Subject field. |
| Certificates | Issuer Name | The Certificate Authority (CA) that issued and signed the certificate. Extracted from the X.509 Issuer field (e.g., "CN=DigiCert Global G2 TLS RSA SHA256 2020 CA1, O=DigiCert Inc"). Full chain extracted: leaf, intermediate, root. |
| Certificates | Not Valid Before | The start date and time from which the certificate is valid. Extracted from the X.509 notBefore field during TLS certificate chain extraction. |
| Certificates | Not Valid After | The expiration date and time of the certificate. Extracted from the X.509 notAfter field. The certificate becomes invalid after this timestamp. Expired certificates are flagged in the CBOM. |
| Certificates | Signature Algorithm Reference | The cryptographic algorithm used to sign the certificate and its OID. Examples: sha256WithRSAEncryption (OID: 1.2.840.113549.1.1.11), ecdsa-with-SHA256 (OID: 1.2.840.10045.4.3.2). PQC signatures (ML-DSA, SLH-DSA, FN-DSA) detected via OID matching per NIST FIPS 204/205/206. |
| Certificates | Subject Public Key Reference | The public key algorithm and size used by the certificate subject. Examples: RSA 2048-bit, RSA 4096-bit, ECDSA P-256 (256-bit), ECDSA P-384 (384-bit), Ed25519 (256-bit). PQC keys (ML-KEM, ML-DSA) detected and labeled "Quantum-Safe". |
| Certificates | Certificate Format | X.509 — all certificates extracted during TLS scans follow the X.509 standard (RFC 5280). This is the most widely used format for TLS/SSL certificates. |
| Certificates | Certificate Extension | .crt or .pem — standard file extensions for X.509 certificates in DER/PEM encoding. SANs (Subject Alternative Names) are also extracted as certificate extensions during scanning. |
